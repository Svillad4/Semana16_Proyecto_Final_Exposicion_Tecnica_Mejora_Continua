# 10 - Guia de clase profunda: exposicion final, argumentacion y mejora continua

## 1. Para que existe esta guia

Esta guia existe para ayudarte a comprender que una entrega final de software no se limita a mostrar una pagina en el navegador. Cuando presentas un proyecto web, debes demostrar tres cosas:

1. Que el proyecto funciona.
2. Que entiendes como fue construido.
3. Que puedes explicar como podria mejorar.

En una clase de programacion, muchas veces se piensa que el trabajo termina cuando el codigo no muestra errores. Pero en un proyecto real, el cierre incluye pruebas, documentacion, evidencias, exposicion, argumentacion y mejora continua.

---

## 2. Que significa exponer un proyecto web final

Exponer un proyecto web final significa presentar de forma organizada la solucion que construiste. No se trata de hablar mucho ni de leer todo el codigo. Se trata de explicar con claridad:

- Que problema querias resolver.
- Que hiciste para resolverlo.
- Que tecnologias usaste.
- Como funciona el proyecto.
- Como comprobaste que funciona.
- Que errores encontraste.
- Que correcciones aplicaste.
- Que decisiones tecnicas tomaste.
- Que mejoras propones para el futuro.

Una buena exposicion combina explicacion, demostracion y reflexion tecnica.

---

## 3. Diferencia entre mostrar y demostrar

Mostrar es simplemente abrir la pagina.

Demostrar es probar una funcionalidad, explicar que deberia pasar, ejecutar la accion y comprobar el resultado.

Ejemplo de mostrar:

> Esta es mi pagina y aqui esta el formulario.

Ejemplo de demostrar:

> Voy a ingresar un correo invalido para comprobar que la validacion funciona. El sistema muestra un mensaje de error. Ahora voy a ingresar datos correctos y el visitante se guarda en JSON Server. Puedo comprobarlo abriendo la ruta `localhost:3001/visitantes`.

La segunda explicacion tiene mas valor porque demuestra dominio del proyecto.

---

## 4. Que significa argumentar decisiones tecnicas

Argumentar significa explicar las razones de una decision. En programacion, no basta con decir que usaste una herramienta. Debes explicar por que la usaste y que beneficio trajo al proyecto.

Ejemplo basico:

> Use Bootstrap.

Ejemplo argumentado:

> Use Bootstrap porque facilita el diseno responsive y permite construir componentes visuales consistentes como botones, formularios, tarjetas y navegacion. Esto mejora la presentacion del proyecto y ayuda a que funcione mejor en celulares.

Una buena argumentacion responde tres preguntas:

1. Que decision tome?
2. Por que la tome?
3. Que beneficio produjo?

---

## 5. Que significa mejora continua

La mejora continua es la capacidad de reconocer que un proyecto puede estar funcionando, pero aun puede crecer y optimizarse.

Un proyecto puede mejorar en muchas areas:

- Seguridad.
- Accesibilidad.
- Usabilidad.
- Rendimiento.
- Diseno visual.
- Organizacion del codigo.
- Base de datos.
- Nuevas funcionalidades.
- Documentacion.
- Pruebas.

La mejora continua no es decir "mejoraria todo". Debes ser especifico.

Ejemplo debil:

> Mejoraria la pagina.

Ejemplo correcto:

> Agregaria autenticacion de usuarios para proteger la informacion registrada. Esto se podria implementar creando una coleccion de usuarios y una pantalla de inicio de sesion. La prioridad seria alta porque protege el acceso a los datos.

## 6. Ruta recomendada para desarrollar la actividad

Sigue esta ruta:

1. Lee el README completo.
2. Instala las librerias con `npm install`.
3. Ejecuta el proyecto con `npm run start`.
4. Abre el sitio web en `localhost:5173`.
5. Abre JSON Server en `localhost:3001`.
6. Prueba el formulario.
7. Revisa la consola.
8. Toma capturas.
9. Completa `README_ENTREGA.md`.
10. Completa el guion de exposicion.
11. Completa decisiones tecnicas.
12. Completa el plan de mejora continua.
13. Ejecuta `npm run check`.
14. Comprime y entrega en Moodle.

---

## 7. Como estudiar antes de exponer

Antes de exponer, no memorices todo. Comprende estas ideas:

- Que hace el proyecto.
- Que problema resuelve.
- Como se ejecuta.
- Donde se guardan los datos.
- Que validaciones existen.
- Que librerias se usan.
- Que evidencias tienes.
- Que mejoras propones.

Si entiendes eso, podras responder preguntas con mas seguridad.

---

## 8. Errores comunes en una exposicion tecnica

Evita estos errores:

1. Abrir el proyecto sin haberlo probado antes.
2. No revisar si JSON Server esta funcionando.
3. Mostrar una consola con errores sin explicar.
4. No tener evidencias.
5. Decir que usaste una herramienta sin saber para que sirve.
6. Leer codigo sin explicar la logica.
7. No justificar decisiones tecnicas.
8. No proponer mejoras futuras.
9. No tener orden en la presentacion.
10. Improvisar completamente.

---

## 9. Que hace que una exposicion sea buena

Una buena exposicion tiene:

- Orden.
- Claridad.
- Demostracion funcional.
- Evidencias.
- Lenguaje tecnico adecuado.
- Seguridad al explicar.
- Reconocimiento de errores corregidos.
- Propuestas de mejora.

No necesitas saberlo todo de memoria. Necesitas demostrar que entiendes tu proceso.

---

## 10. Cierre

Esta actividad te prepara para presentar un proyecto como desarrollador: no solo entregas archivos, sino una solucion que puedes ejecutar, probar, explicar y mejorar.
